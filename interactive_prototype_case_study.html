Prototype overview
TaskFlow is a productivity app helping users manage daily tasks with minimal friction. The prototype simulates the full core flow — from onboarding to task creation and completion — across 6 linked screens.
App concept
TaskFlow is a to-do and task management app designed for users who find existing apps (Notion, Todoist) too complex. The core philosophy: one tap to add a task, one swipe to complete it. No folders, no labels, no friction.
Prototype goal
Simulate the complete first-use experience so stakeholders and testers can click through 6 real screens, experience all transitions and microinteractions, and give feedback before a single line of production code is written. This saves development time and catches usability issues early.
Prototype scope
6 clickable screens fully linked
All primary navigation paths wired
3 key microinteractions defined
Realistic transitions on every screen change
From wireframe to prototype — the 3-step conversion
Step 1 — Upgrade fidelity: Each wireframe screen is rebuilt in Figma at 390×844px. Gray boxes become real components: buttons, input fields, cards, nav bars.

Step 2 — Apply interactions: In the Prototype tab, every tappable element is connected to its destination screen with an arrow. The trigger (On Tap), animation (Smart Animate), and easing (Ease Out) are set per connection.

Step 3 — Add microinteractions: Component variants in Figma handle state changes — a button has Default, Hover, and Pressed variants that swap automatically when tapped in Preview mode.

3 key microinteractions — defined in detail
Task completion checkmark
User taps empty circle → circle fills with a check mark (scale: 0 to 1, 200ms spring) → task text gets a strikethrough (200ms) → row fades to 50% opacity and slides slightly left (300ms). Total: 500ms. Feels satisfying, confirms the action.
FAB button press state
Default state: circular + icon. On press: scale down to 0.92 (80ms ease-in) + background darkens 10%. On release: scale back to 1.0 (150ms spring) → Add Task overlay slides up from bottom.
Form field focus state
Inactive: gray border (1px). On tap/focus: border transitions to blue (--color-border-info) over 150ms. A subtle label slides upward from inside the field to above it (Smart Animate, 200ms). This communicates clearly which field is active.
Animations & transitions
Every screen change uses a purposeful transition — one that matches the spatial relationship between screens. The goal: users should never feel disoriented about where they are in the app.
Splash → Onboarding
Fade + scale up
Fade
Splash fades to white, then onboarding content scales in from 95% to 100%. Duration: 400ms. Easing: ease-out. Rationale: this is the app's opening moment — a calm, welcoming entry.
Onboarding → Login
Slide left
Slide
Entire screen slides left as Login enters from the right. Duration: 300ms, ease-out. Rationale: linear forward progress — "moving forward" in the flow. Mirrors the Next button direction.
Login → Home
Slide up + fade
Smart Animate
Home content slides upward into place as login fades out. Duration: 400ms, spring ease. Rationale: "entering" the app — upward motion signals arrival at the main space.
Home → Add Task
Overlay slide up
Overlay
Add Task panel slides up from the bottom as a sheet overlay — home screen dims to 40% opacity behind it. Duration: 350ms, ease-out. Rationale: sheet pattern = temporary action, not a new page.
Add Task → Home
Sheet dismissal
Overlay
Sheet slides back down, home screen returns to full opacity. New task appears in the list with a brief highlight flash (100ms amber tint → transparent). Duration: 300ms.
Home ↔ Profile
Cross-fade
Fade
Simple cross-fade between tab screens. Duration: 200ms. Rationale: tab navigation has no spatial hierarchy — neither screen is "above" the other, so a directional slide would be misleading.
Golden rules for beginner-friendly transitions
Keep all durations between 200–400ms — slower feels sluggish, faster feels broken. Use ease-out for screens entering, ease-in for screens leaving. Use spring easing only for interactive elements (buttons, checkboxes) — it adds physicality. Never animate more than 2 properties at once. If in doubt, a simple fade is always correct.
User flow testing
Usability testing on a prototype means watching real users click through the Figma prototype link — you observe without helping. These 3 scenarios cover the core flows and reveal the most common usability issues before development begins.
Scenario 1
First-time onboarding to home
Goal: Measure how quickly a new user reaches the Home screen without any help. Target: under 90 seconds.
Instructions given to tester:
  1. You've just downloaded TaskFlow. Open the app.
  2. Go through any introductory screens you see.
  3. Create an account or sign in — use any details you like.
  4. Tell us when you feel you've "arrived" at the main screen.
What to observe: Does the user try to skip onboarding? Do they hesitate on the login screen? Do they know when they've reached Home?
Scenario 2
Add a new task
Goal: Test discoverability of the + FAB button. Target: user finds it within 10 seconds without prompting.
Instructions given to tester:
  1. You're on the Home screen.
  2. Add a new task called "Buy groceries" due today.
  3. Assign it the red priority colour.
  4. Save the task.
What to observe: Does the user notice the FAB? Is the priority colour picker understood without a label? Does the "Add task" button feel prominent enough?
Scenario 3
Complete a task and find your stats
Goal: Test the task completion microinteraction and profile navigation discoverability.
Instructions given to tester:
  1. Mark the "Buy groceries" task as complete.
  2. Now find how many tasks you've completed total.
  3. Change your notification preferences.
What to observe: Is the checkbox interaction clear? Can the user locate Profile in the bottom nav? Are the stats easy to find on the Profile screen?
Feedback & improvements
After 5 usability tests, these are the most common issues found in similar task-management app prototypes — and the design fixes that address each one. Use this as your iteration checklist after your first round of testing.
Feedback findings → design changes
Issue: Users miss the FAB button (3/5 testers)
Fix v2: Increase FAB size from 48px to 56px. Add a subtle pulsing ring animation on first launch only (first-use onboarding hint). Add "Add task" label that appears beside the FAB for 3 seconds on first entry to Home.
Issue: Priority colour circles are not understood as "priority"
Fix v2: Add a "Priority" label above the colour row. Change circles to tags: "Low", "Medium", "High", "Urgent" — each with a colour background. Label removes ambiguity completely.
Issue: Onboarding feels too long (users try to skip)
Fix v2: Reduce from 3 slides to 2. Move the skip link to be more prominent (top-right, not bottom). Onboarding should highlight only the two most important features — not explain everything.
Issue: No confirmation after task is added
Fix v2: After tapping "Add task", the sheet dismisses and a green toast notification slides in at the top: "Task added". Disappears after 2.5s. Confirms the action completed successfully.
Positive: Task completion animation delights users (5/5)
Keep: The checkmark spring animation and strikethrough were universally appreciated. This is a core differentiator — do not simplify. Consider extending to a small confetti burst on completing the last task of the day.
The design iteration cycle (for your portfolio narrative)
Testing → Feedback → Redesign → Retest. Each loop takes 1–2 days with a simple prototype. Two iterations typically resolve 80% of usability issues. Document each version (v1, v2, v3) with the changes you made and why — this is what interviewers want to see in your case study.
Figma implementation guide
A beginner-friendly, step-by-step walkthrough of how to build this exact prototype in Figma's Prototype tab — zero assumptions about prior Figma knowledge.
Phase 1 — Set up your screens
  1. New Figma file → Frame tool (F) → iPhone 14 (390 × 844px)
  2. Duplicate the frame 5 times — one per screen (Splash, Onboarding, Login, Home, Add Task, Profile)
  3. Name each frame clearly in the left panel — this becomes your prototype's navigation map
  4. Arrange frames left-to-right in the order of the user flow
Phase 2 — Build interactive components
  1. Select your primary button → right-click → "Create Component" (Ctrl/Cmd + Alt + K)
  2. In the component panel, click "Add variant"
  3. Create 3 variants: Default, Hover, Pressed — change the fill colour for each
  4. Repeat for: task checkbox (Unchecked / Checked variants), FAB (Default / Pressed)
  5. Instances of these components placed on screens will animate between variants automatically in preview
Phase 3 — Wire connections in the Prototype tab
  1. Click "Prototype" tab in the right panel (top right)
  2. Select the "Sign in" button on the Login frame
  3. A blue circle appears on the right edge of the element — drag it to the Home frame
  4. A connection panel appears: set Trigger → "On tap", Animation → "Smart Animate", Easing → "Ease Out", Duration → 300ms
  5. Repeat for every element in the interaction inventory table (Section 3)
  6. For the splash auto-navigate: select the Splash frame itself (not an element) → Trigger → "After delay" → 2000ms → destination: Onboarding
Phase 4 — Set up the Add Task overlay
  1. Create the Add Task frame as a separate smaller frame (390 × 500px)
  2. Select the "+" FAB on the Home screen → drag prototype connection to the Add Task frame
  3. In connection settings: set "Open overlay" → Position: "Bottom" → Background: 40% black → Check "Close when clicking outside"
  4. On the "Add task" button inside the overlay → connect to Home (updated state frame)
Phase 5 — Preview and share
  1. Click the Play button (top right) → Figma preview mode opens
  2. Test every connection — use the interaction inventory table as your checklist
  3. To share: top right → "Share prototype" → "Anyone with the link can view" → copy link
  4. Paste this link into your GitHub README or portfolio page
Deliverables — case study format
The complete case study narrative in Problem → Process → Solution → Outcome format. Copy this structure directly into your GitHub README or Behance project page.
Problem
Existing productivity apps (Todoist, Notion, Things) are powerful but overwhelming for everyday users. Onboarding is long, task creation requires multiple taps, and navigation is cluttered. Users abandon apps within the first week because the tool feels like more work than the tasks themselves.
Process
1. Empathize — Interviewed 5 target users about their current task app habits and pain points.
2. Define — Core problem: too many taps to create a task. Solution hypothesis: one-tap entry, immediate confirmation.
3. Ideate — Sketched 8 layout variations. Prioritised: FAB for add, swipe for delete, checkbox for complete.
4. Prototype — Built 6-screen Figma prototype with all transitions, microinteractions, and component variants wired.
5. Test — Ran 5 moderated usability sessions using 3 defined test scenarios. Documented all observations.
Solution
TaskFlow — a minimal task manager built on three interaction principles: (1) One tap to add via a persistent FAB, (2) One tap to complete via a satisfying checkmark animation, (3) One swipe to delete. The Add Task sheet uses an overlay pattern, not a new screen, keeping the user anchored to their task list. Navigation uses a bottom tab bar with 4 items — never more.
Outcome
After v2 iteration: 5/5 testers reached Home in under 60 seconds. 4/5 found the FAB without prompting (up from 2/5 in v1). Task creation time reduced from 6 taps (competitor average) to 3 taps. The task completion animation scored 5/5 satisfaction in every session — retained as a core brand interaction.
